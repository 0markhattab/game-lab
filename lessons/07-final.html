Module 07 · Capstone

Build Geometry Dash

Time to combine everything. The world scrolls. A cube auto-runs. Spikes try to kill it. You hit space at exactly the right moment to jump. We'll build it in five passes — each pass adds one layer of polish until it's a real, playable, satisfying game.

07.0

Architecture overview

Before any code, here's the mental model. The cube doesn't actually move forward — it stays at x = 80. It's the world that scrolls past. This is how every endless runner works under the hood.

// Each frame:
update_player_physics();    // gravity, jump
scroll_world_left();        // move all obstacles by -speed
spawn_new_obstacles();      // when needed
remove_offscreen();         // cleanup
check_collisions();         // die if you hit a spike
draw();

Pass 1: cube + ground + jump

Start small. Just a cube on a ground that can jump. We've done this before in Lesson 05 — copy that mental model in.

Why does the cube rotate while airborne? Pure visual polish — without it, the cube looks lifeless. Robert Topala (Geometry Dash's creator) noticed players felt the game more when the cube tumbled. One line of code, transforms the feel.
07.2

Pass 2: scrolling world + spikes

Now we make it endless. Add an array of spikes. Each frame, all spikes shift left by the world speed. When a spike falls off the left edge, drop it from the array. When the most recent spike has been on screen long enough, spawn a new one.

07.3

Pass 3: collision + death + restart

Now the cube actually can die. Add an AABB check between the player and each spike's bounding box. On hit, set state = 'dead'. Press R to restart.

pass-3.js
pass-3 · click + space + R to restart● LIVE
The forgiveness margin (m = 4). The visible cube is 28 pixels but its hitbox is 8 pixels smaller. Why? Because feeling cheated when "I clearly cleared that!" is the #1 reason people quit hard games. Shrinking the hitbox by ~15% feels honest while making the game vastly more enjoyable. Every great platformer does this.
07.4

Pass 4 & 5: polish ideas

The game is functionally complete. From here, polish is what separates a school project from a game people play. Here's a menu of what to add — each one is a tiny addition with outsized impact.

Particle trail

Spawn 1-2 small fading squares behind the cube each frame. Push them into an array, decay their alpha, remove dead ones. ~15 lines.

Death explosion

On hit, spawn 12 particles flying outward in random directions. Pause the world for ~30 frames before showing the death screen.

Speed ramps

Increase worldSpeed by 0.001 per frame. Slowly the game gets harder. The score becomes the difficulty.

Audio sting

Use the Web Audio API to play a 100ms beep on jump and a 300ms low tone on death. Two functions, infinite game-feel improvement.

The full polished version with all of this lives at games / geometry-dash. Open it, study the source, then build your own variant in the Studio.

07.5

You did it.

You started with let x = 0 and a frame loop. You're ending with a complete game built from physics, input, collision, and state management — the same building blocks every game studio uses to ship million-dollar titles. The tools haven't changed since Pong; only what you decide to build with them.
